# Technical Debt

- **pnpm db:generate**: CJS/ESM resolution broken on Windows with drizzle-kit — migrations must be written manually. (discovered in #271)
- **CSP unsafe-inline**: `'unsafe-inline'` was added to scriptSrc in helmet config to allow the injected `window.__NARRATORR_URL_BASE__` config script. Consider using a nonce-based approach instead for stricter CSP. (discovered in #284)
- **registerStaticAndSpa untested**: The HTML injection and SPA fallback logic in `server-utils.ts` isn't unit tested because it requires a real filesystem with `dist/client/index.html`. An integration test with a temp directory would be more robust. (discovered in #284)
- **QualityGateService max-lines override**: File grew to 535 lines with SSE emissions, requiring an eslint-disable for max-lines. The SSE emission calls are mechanical boilerplate — consider extracting a helper that takes (broadcaster, eventType, payload) to reduce line count. (discovered in #283)
- **matchesStatusFilter missing 'failed' status**: `matchesStatusFilter()` in `helpers.ts` doesn't include 'failed' in any filter tab — failed books are invisible in all non-'all' tabs. Should either be included in 'downloading' tab or get its own tab. (discovered in #282)
- **Activity/event-history routes still use string-matching error handling**: The approve, reject, retry, and mark-failed routes match error messages via `message.includes('not found')` etc. instead of using typed error classes. Needs typed errors in QualityGateService, retrySearch, and EventHistoryService before these can delegate to the global error handler. (discovered in #359)
- **No backup encryption**: Backups contain the raw SQLite DB including hashed passwords and API keys. For a self-hosted single-user app this is acceptable, but if multi-user support is ever added, backups should support optional encryption. (discovered in #280)
- **GeneralSettings probeFfmpeg untested during form submission**: The mutation in `GeneralSettings.tsx` calls `probeFfmpeg` before `updateSettings` when processing is enabled, but no test covers this path or the error case where `probeFfmpeg` rejects. The probe button in `ProcessingSettingsSection` tests the endpoint directly but not through the form submission flow. (discovered in #332)
- **Pending restore state is in-memory only**: The `pendingRestore` object with 5-min TTL lives in BackupService memory. If the server restarts between upload and confirm, the pending state is lost. Low risk since the TTL is short, but a file-based marker would be more resilient. (discovered in #280)
- **SettingsService.update() doesn't deep-merge**: `update(partial)` calls `set()` per category which overwrites the full category value. Code that needs to update a single field must `get()` + spread + `set()` manually. Should either add a `patch(category, partialFields)` method or make `update()` deep-merge. (discovered in #333)
- ~~**api-collision.test.ts missing backupsApi**: Resolved in #358 — collision guard now covers all 18/18 modules.~~
- **CredentialsSection lacks dedicated test file**: The component is tested indirectly through SecuritySettings integration tests, but has no co-located unit test. Direct failures in form submission, error handling, or state management could go undetected. (discovered in #358)
- **TaskRegistry.estimateNextRun() is a rough approximation**: Only handles simple `*/N` cron patterns. A proper cron parser library (like `cron-parser`) would give accurate next-run times for complex expressions like `0 0 * * 0`. (discovered in #279)
- **system.ts hardcodes version '0.1.0'**: The `/api/system/info` route returns a hardcoded version string. Should read from `package.json` at startup. (discovered in #279)
- **CI quality-gates job duplicated**: `docker.yaml` copies the full quality-gates job from `ci.yaml` because Gitea Actions doesn't support reusable workflows. If a third workflow appears, consider extracting a composite action. (discovered in #175)
- **RecyclingBinService restore doesn't handle concurrent restores**: Two simultaneous restore calls for the same entry could both pass the conflict check and race on the filesystem move. Low risk since it requires deliberate double-click past the UI's disabled state, but a DB-level lock or optimistic concurrency check would be more robust. (discovered in #331)
- **tailwind-merge unused dependency**: `tailwind-merge` is listed in `package.json` but never imported anywhere in source code. Should be removed if not needed, or used if there's a cn() utility planned. (discovered in #329)
- **eslint-plugin-react-hooks blocks ESLint 10**: Peer dep only goes up to ESLint 9. Need to wait for a stable release with ESLint 10 support before upgrading. (discovered in #329)
- **scripts/metrics.ts unused imports**: `gitea` and `parseLabels` are imported but never used. Suppressed with underscore aliases to pass typecheck. Should either be used or removed. (discovered in #315)
- **No key rotation CLI**: Encryption key changes are unsupported — changing the key requires manual re-encryption of all secrets. A `narratorr key-rotate` CLI command would automate decrypt-with-old/encrypt-with-new. (discovered in #315)
- **ImportListsSettings form labels lack htmlFor**: The form in `ImportListsSettings.tsx` uses `<label>` elements without `htmlFor`/`id` pairing, making them inaccessible to screen readers and breaking `getByLabelText` in tests. Settings fields use placeholder text as the only test anchor. (discovered in #285)
- ~~**claim.ts doesn't handle existing branches**: Resolved in #353 — `findExistingBranch()` checks local/remote branches and resumes them.~~
- **Hardcoded processing fixtures in test files**: Multiple test files (`BookDetails.test.tsx`, `ProcessingSettingsSection.test.tsx`) use full hardcoded processing objects instead of `createMockSettings()` with partial overrides. Every new processing field requires touching 10+ fixtures. These should use partial overrides to inherit new defaults automatically. (discovered in #198)
- **claim.ts doesn't check for unmerged files**: Pre-existing merge conflict markers from `git stash pop` block ALL commits on every branch. `claim.ts` creates a new branch but doesn't check `git status` for UU (unmerged) files first — should warn or auto-resolve before proceeding. (discovered in #362)
- **runUpgradeSearchJob not deduplicated**: `runUpgradeSearchJob` in `jobs/search.ts` still has its own search→filter→rank→grab loop with quality comparison logic that couldn't be consolidated into `searchAndGrabForBook` (it needs `compareQuality` + double grab-floor check). A separate `searchAndGrabUpgrade` helper could DRY this up, but it was out of scope for #357. (discovered in #357)
- **jobs/search.ts re-exports from search-pipeline**: `searchAndGrabForBook` and `SingleBookSearchResult` are re-exported from `jobs/search.ts` for backward compatibility. These re-exports can be cleaned up when all consumers are updated to import directly from `services/search-pipeline.ts`. (discovered in #357)
- **BookService.getAll() slim select**: The slim select uses an explicit column list that must be kept in sync with the books table schema. Adding a new column to the books table requires updating the slim select field list in `book.service.ts`. Consider deriving the slim select from the full table definition minus excluded fields. (discovered in #355)
- **importDownload() still has complexity 42 / 160 lines**: The hard-fail pipeline (source validation, disk space check, file copy, audio processing, copy verification) accounts for most of the remaining complexity. Extracting these as throwing helper functions (like `validateSource`, `checkDiskSpace`, `copyToLibrary`) would eliminate the per-function eslint suppression entirely. Out of scope for #349 (best-effort phases only). (discovered in #349)
