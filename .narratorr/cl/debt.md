# Technical Debt

- **`src/server/utils/paths.ts` / `src/server/utils/import-helpers.ts` / `src/server/services/library-scan.service.ts`**: `extractYear()` is now tripled — library-scan.service.ts added a third variant for folder-name year extraction. All three should be consolidated into a single shared utility (discovered in #210, worsened in #235)
- **`src/client/pages/settings/LibrarySettingsSection.tsx`**: ~~At exactly 400 lines~~ **RESOLVED in #212** — reduced to ~105 lines after extracting NamingSettingsSection
- **`src/shared/schemas/settings/library.ts`**: Schema refinement chains (hasTitle + validateTokens) are still duplicated inline between folderFormatSchema/fileFormatSchema and libraryFormSchema because Zod v4's type system doesn't allow generic refinement wrapper functions that satisfy ZodObject shape constraints. Could be revisited if Zod v4 adds better type utilities (discovered in #212)
- **`src/shared/schemas/settings/strip-defaults.ts`**: `stripDefaults()` loses TypeScript field types — returns `z.ZodObject<Record<string, z.ZodType>>` instead of preserving the original shape. Categories needing `.pick()` or precise `z.infer` types must use explicit form schemas. A type-preserving `stripDefaults<T>()` generic would eliminate this duplication (discovered in #215)
- **`src/client/pages/settings/ProcessingSettingsSection.tsx`**: ~~`as any` cast on zodResolver~~ **RESOLVED in #219** — replaced `z.preprocess` with `setValueAs` in `register()` call
- **`src/shared/schemas/settings/processing.ts`**: Shared `processingFormSchema` still uses `z.preprocess(nanToUndefined, ...)` which has the same input/output type divergence. Not blocking (used by registry, not zodResolver directly), but should be aligned with the `setValueAs` approach if the shared schema ever needs zodResolver compatibility (discovered in #219)
- **`src/client/components/settings/` inputClass duplication**: `inputClass` / `errorInputClass` constants are still copy-pasted across BlackholeFields, DownloadClientFields, NotifierFields, CredentialsSection, and ImportList files. These could be extracted to a shared `formStyles.ts` alongside `SelectWithChevron` (discovered in #216)
- **`src/client/pages/settings/ImportListProviderSettings.tsx`**: No co-located test file. Provider-specific settings (ABS library fetch, NYT list select, Hardcover conditional shelf ID) are only covered indirectly via the parent `ImportListsSettingsSection.test.tsx` (discovered in #216)
- **`src/shared/schemas/settings/library.ts`**: `hasTitle()` and `hasAuthor()` use independent hardcoded regexes rather than deriving from `TOKEN_PATTERN_SOURCE`. When the token grammar changes (e.g., adding prefix syntax), these must be manually updated in parallel. Could be derived from TOKEN_PATTERN_SOURCE to stay in sync automatically (discovered in #228)
- **`src/client/pages/library/FilterRow.tsx` + `src/client/components/manual-import/ImportSummaryBar.tsx`**: 4 raw `<select>` elements still use manual `appearance-none` + `ChevronDownIcon` pattern instead of the shared `SelectWithChevron` component. These were explicitly out of scope for #224 but should be migrated for consistency (discovered in #224)
- **`src/client/pages/manual-import/PathStep.tsx`**: No co-located test file. Date formatting and folder history rendering are only covered indirectly via ManualImportPage.test.tsx (discovered in #224)
- **`src/client/components/ManualAddForm.tsx`**: `seriesPosition` conversion uses falsy check (`data.seriesPosition ? Number(...) : undefined`) which treats "0" as falsy. Should use `data.seriesPosition !== '' && data.seriesPosition !== undefined` for correctness if position 0 is valid. Low priority — series positions conventionally start at 1 (discovered in #246)
- **`src/core/utils/audio-processor.ts`**: `convertFiles()` (lines 217-224) unconditionally injects `trackNumber`, `trackTotal`, `partName` even for single-file non-`.m4b` inputs routed through `processAudioFiles()`. The intermediate filenames get track suffixes that are unnecessary for single-file books. `renameFilesWithTemplate()` was fixed in #231 but `convertFiles()` was intentionally left out of scope (discovered in #231)
