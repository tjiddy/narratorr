### F1: Prowlarr config endpoints bypass masked-secret/sentinel contract
**What was caught:** GET /api/prowlarr/config returned the decrypted apiKey; PUT just re-saved whatever string came in without sentinel handling.
**Why I missed it:** The /plan step focused on services and the shared CRUD pattern. Prowlarr routes are standalone (not registered via `registerCrudRoutes`), so they were missed during the masking integration sweep.
**Prompt fix:** Add to /plan's explore step: "When adding cross-cutting route behavior (masking, auth, validation), grep ALL route files for endpoints that return the affected entity types — not just routes using shared helpers."

### F2: Autogenerated encryption keys created silently
**What was caught:** `loadEncryptionKey()` returned a Buffer but didn't indicate whether a new key was generated, so startup had no audit trail.
**Why I missed it:** Focused on the crypto correctness and overlooked the observability requirement in AC6.
**Prompt fix:** Add to /implement's self-review checklist: "For each new startup-path function: does the caller have enough information to log which code path was taken? Security-sensitive paths (key generation, migration) must be loggable."

### F3/F4: Missing route-level masking tests for indexers and download clients
**What was caught:** No assertions that secret fields come back as `'********'` in API responses.
**Why I missed it:** The coverage review subagent flagged this but I was focused on fixing typecheck/lint issues and proceeded to handoff without addressing the gap.
**Prompt fix:** Add to /handoff coverage review: "For each new response transformation (masking, filtering, formatting), at least one route test must assert the transformation's output — testing the service layer alone is insufficient."

### F5: Sentinel proxy URL triggers false cache invalidation
**What was caught:** Comparing `data.network` (with `proxyUrl: '********'`) against decrypted `previousNetwork` always shows a diff, clearing the adapter cache on every save.
**Why I missed it:** The interaction between masking and change-detection was a predictable consequence I didn't trace. The self-review step should have caught this — it's a classic "feature A interacts with feature B" scenario.
**Prompt fix:** Add to /handoff self-review interaction analysis: "When adding response masking to fields that also participate in change-detection logic (dirty checks, cache invalidation, diff comparisons), verify that sentinel values are normalized BEFORE the comparison."

### F6: SettingsService network encryption/decryption/sentinel untested
**What was caught:** No service-level tests for the network category encryption, decryption, or sentinel passthrough.
**Why I missed it:** The SettingsService tests were pre-existing and I only added the encryption code to the service without adding corresponding tests. The coverage subagent flagged it but was dismissed as a false negative.
**Prompt fix:** Add to /implement: "When modifying a service method's behavior (not just adding a new method), always add tests for the new behavior in the existing test file — even if the method was already 'tested' before your change."
